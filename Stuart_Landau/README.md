# PPO Hybrid SOS/ZPK for Stuart–Landau Oscillator

This repository contains a hybrid **MATLAB + Python** pipeline for stabilizing the nonlinear Stuart–Landau oscillator using dynamic controllers represented in **SOS/ZPK** format and optimized with **PPO**.

The main objective is to start from an initial controller `K0`, synthesized in MATLAB, convert it into a discrete second-order-section representation, import it into Python, optimize its parameters using PPO, and finally validate the resulting controller `KPPO` again in MATLAB/Simulink.

---

## 1. General Pipeline

The complete workflow is divided into three main stages:

1. **MATLAB**
   - Synthesize an initial controller `K0` using LQG.
   - Convert `K0` into a discrete SOS/ZPK representation.
   - Export the initial controller as a `.mat` file.

2. **Python**
   - Import the initial controller `K0`.
   - Simulate the nonlinear Stuart–Landau dynamics.
   - Train PPO over the SOS/ZPK controller parameters.
   - Export the final controller `KPPO` and closed-loop signals.

3. **MATLAB / Simulink**
   - Load the files generated by Python.
   - Simulate the closed-loop system with `K0` and `KPPO`.
   - Compare MATLAB and Python responses.
   - Validate the complete SOS/ZPK + PPO pipeline.

## 2. Main Auxiliary Functions

### `K0_lqg_synth`

This function synthesizes an initial LQG controller for the linearized Stuart–Landau dynamics around the design point `mu_design`.

This initial controller is used as the starting point for the PPO pipeline.

The LQR and observer weights are defined internally by default.

**Main inputs:**

- `mu_design`: parameter used to build the linear design model.
- `w`: base frequency of the Stuart–Landau oscillator.

**Main output:**

- `K0_lqg`: initial LQG controller in state-space form.

This controller is then passed to `K0_sos2zpk`.

---

### `K0_sos2zpk`

This function converts an initial controller `K0` into a structure compatible with the Python SOS/ZPK implementation.

The function accepts controllers in the following formats:

- `ss`
- `tf`
- `zpk`

It performs the following steps:

1. Discretizes the continuous-time controller using `c2d`.
   - Default method: `zoh`.

2. Converts the controller into second-order sections:
   - `ss2sos` if the controller is given in state-space form.
   - `tf2sos` if the controller is given as a transfer function.
   - `zp2sos` if the controller is given in zero-pole-gain form.

3. Organizes and normalizes the SOS terms.

4. Extracts:
   - poles per section,
   - zeros per section,
   - local denominator order,
   - local numerator order,
   - global gain,
   - sampling time.

5. Exports the initial controller as a `.mat` file for Python.

The generated file contains:

```matlab
P_sections
Z_sections
pole_order
zero_order
Kg
Ts
K0_sos_raw
K0_sos_norm
```
The main generated file is `controller_K0_SOS_ZPK.mat`. 

## 3. Step 1: Initial `K0` Synthesis in MATLAB

The first MATLAB block synthesizes the initial controller `K0`.

Main parameters:

```matlab
w = 2;
beta = 0.5;
mu_design = -0.5;
mu_target = 4;
x0 = [0.6; 0.1];
dt = 2e-2;
Ts = 2e-2;
```

### Parameter Meaning

| Parameter | Description |
|---|---|
| `mu_design` | Value of `mu` used to design the initial LQG controller on the linear Stuart–Landau model. |
| `mu_target` | Value of `mu` to be stabilized in the nonlinear Stuart–Landau dynamics. |
| `dt` | Physical integration / sampling step used for simulation and plotting. |
| `Ts` | Sampling time of the discrete controller. |
| `x0` | Initial condition of the nonlinear Stuart–Landau dynamics. |

The linear model used for the initial controller synthesis is:

```matlab
A = [mu_design -w;
     w          mu_design];

B = [1; 0];
C = [0, 1];
D = zeros(size(C,1),1);
```

The LQG controller is synthesized as:

```matlab
K0_lqg = K0_lqg_synth(mu_design, w);
```

Then it is converted into the discrete SOS/ZPK format:

```matlab
K0_zpk = K0_sos2zpk(K0_lqg, Ts, 'ss', save_dir_sos);
```

The main output of this stage is:

```text
controller_K0_SOS_ZPK.mat
```

This file is used directly by the Python PPO pipeline.

---

## 4. Step 2: PPO Training in Python

Once `controller_K0_SOS_ZPK.mat` has been created, the Python script is executed.

The purpose of the Python stage is to:

1. Load the initial controller `K0`.
2. Reconstruct it as a dynamic SOS/ZPK controller.
3. Simulate the nonlinear Stuart–Landau dynamics.
4. Train PPO to modify the controller parameters.
5. Export the final controller `KPPO`.

The controller imported from MATLAB is used as the initial base controller:

```text
K0 -> SOS/ZPK -> Python DynamicControllerSOS
```

During training, PPO does not directly act on the physical input force. Instead, PPO modifies the controller parameters.

---

## 5. PPO Environment Semantics

In this pipeline:

```text
one env.step(action) == one physical/solver step
```

This means that each environment step corresponds to one physical simulation step of the nonlinear Stuart–Landau dynamics.

PPO does not directly propose the physical control force `u[k]`.

Instead, PPO proposes a perturbation over the controller parameters:

```text
$theta_{trial} = theta_{base} + \delta_{theta}$
```

Then a dynamic SOS/ZPK controller is reconstructed:

```text
K_trial = DynamicControllerSOS(theta_trial)
```

The controller generates the physical control input from the measured output:

```text
y = Csens @ xp
u = K_trial.sample(y)
```

Therefore, the physical closed-loop simulation is performed with the dynamic controller generated from the PPO-proposed parameter vector.

## 6. Important Python Parameters

The main training parameters are defined in the Python script:

```python
MU_TARGET = 4
DT = 2e-2
TS = 2e-2
MAX_STEPS = 50
HYBRID_TIMESTEPS = 50000
PPO_N_STEPS = 1024
BATCH_SIZE = 64
DK_HOLD_STEPS = 50
K_EVAL_STEPS = 500
```

### Description

| Parameter | Description |
|---|---|
| `MU_TARGET` | Value of `mu` for the nonlinear Stuart–Landau dynamics to be stabilized. |
| `DT` | Physical integration step of the nonlinear plant. |
| `TS` | Sampling time of the discrete controller. |
| `MAX_STEPS` | Number of physical steps per training episode. |
| `MAX_STEPS * DT` | Physical training horizon per episode. |
| `HYBRID_TIMESTEPS` | Total number of physical environment steps used by PPO. |
| `PPO_N_STEPS` | Number of collected steps before each PPO policy update. |
| `BATCH_SIZE` | PPO batch size. |
| `DK_HOLD_STEPS` | Number of physical steps during which the same effective controller proposal is held. |
| `K_EVAL_STEPS` | Number of physical steps used to evaluate a candidate controller. |
| `K_EVAL_STEPS * DT` | Physical evaluation horizon for the candidate controller. |

The most important semantic point is:

```text
total_timesteps = number of physical environment steps
```

and:

```text
one env.step(action) = one physical/solver step
```

## 7. Running the Python Script

Example training command:

```bash
python PPO_HYBRID_SOS_ZPK_STUART_LANDAU.py \
    --mat-file controller_K0_SOS_ZPK.mat \
    --mode train \
    --total-timesteps 50000 \
    --max-steps 50 \
    --dk-hold-steps 50 \
    --eval-final-time 10.0
```

Debug mode:

```bash
python PPO_HYBRID_SOS_ZPK_STUART_LANDAU.py \
    --mat-file controller_K0_SOS_ZPK.mat \
    --mode debug
```

The training mode performs:

1. Initial closed-loop rollout with `K0`.
2. PPO training over the SOS/ZPK controller parameters.
3. Final closed-loop rollout with `KPPO`.
4. Export of the final controller and simulation signals.

## 9. Step 3: MATLAB Post-Processing

After running Python, MATLAB is used again to load the generated `.csv` and `.mat` files.

```matlab
KPPO_FINAL = load("stuart_landau_final_controller_KPPO.mat");
PYTHON_MU_TARGET_K0_csv = readtable("stuart_landau_initial_closed_loop_K0_signals.csv", ...
    VariableNamingRule='preserve');
PYTHON_MU_TARGET_KPPO_csv = readtable("stuart_landau_final_closed_loop_KPPO_signals.csv", ...
    VariableNamingRule='preserve');
```

The final controller `KPPO` is reconstructed in MATLAB as:

```matlab
Kg_ppo = KPPO_FINAL.Kg_final;
P_ppo = KPPO_FINAL.P_sections_final;
Z_ppo = KPPO_FINAL.Z_sections_final(~isnan(KPPO_FINAL.Z_sections_final));
tf_ppo = zpk(Z_ppo, P_ppo, Kg_ppo, Ts);
[A_KPPO, B_KPPO, C_KPPO, D_KPPO] = zp2ss(Z_ppo, P_ppo, Kg_ppo);
```

This allows the final PPO controller to be simulated again in MATLAB/Simulink using the same closed-loop logic.

## 11. Final Validation Blocks

The final MATLAB post-processing performs three main validation blocks.

---

### 11.1. Pole-Zero Map

The poles and zeros of the initial controller `K0` are compared against the poles and zeros of the final controller `KPPO`.

```matlab
[p1, z1] = pzmap(K0_disc);
[p3, z3] = pzmap(tf_ppo);
```

The pole-zero map includes:

- poles of `K0`,
- zeros of `K0`,
- poles of `KPPO`,
- zeros of `KPPO`,
- unit circle.

The goal is to observe how PPO modifies the dynamic structure of the controller.

---

### 11.2. Closed-loop Output Comparison

The output signals `y(t)` are compared for:

1. MATLAB SOS with `K0`.
2. Python SOS with `K0`.
3. MATLAB ZPK with `KPPO`.

The generated figure is:

```text

```

This comparison verifies:

- consistency between MATLAB and Python,
- validity of the SOS/ZPK conversion,
- dynamic improvement obtained with `KPPO`,
- correctness of the complete closed-loop pipeline.

### 11.3. Saturated Force Comparison

## 14. Validation Objective

The purpose of the pipeline is to verify that:

1. The initial controller `K0` designed in MATLAB can be correctly exported to Python in SOS/ZPK format.
2. The dynamic SOS/ZPK controller implementation in Python reproduces the MATLAB response.
3. PPO can modify the SOS/ZPK controller parameters to improve stabilization of the nonlinear Stuart–Landau dynamics.
4. The final controller `KPPO` can be imported back into MATLAB and simulated using the same closed-loop logic.
5. The MATLAB/Python comparison validates the complete input/output consistency of the pipeline.

The validation is based on:

- closed-loop output comparison,
- saturated input force comparison,
- pole-zero map comparison,
- MATLAB/Python response consistency.
